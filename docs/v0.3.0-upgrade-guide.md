## Region Key Normalization (Breaking Change)

Starting in v0.3.0, the module normalizes all `for_each` keys derived from region strings to lowercase AWS format (`us-east-1`). Previously, the module used the raw user-provided region string as the key.

### Who is affected?

Only users who passed **Atlas-format regions** (`US_EAST_1`) to these variables:
- `privatelink_endpoints[].region`
- `encryption.private_endpoint_regions`

If you already use lowercase AWS format (`us-east-1`) in your configuration, **no action is required**.

### How to check

After upgrading the module source to v0.3.0, run `terraform plan`. If you see Terraform destroying and recreating resources because of the region key format change, the update affects your configuration:

```
# module.atlas_aws.mongodbatlas_privatelink_endpoint.this["US_EAST_1"] will be destroyed
# module.atlas_aws.mongodbatlas_privatelink_endpoint.this["us-east-1"] will be created
```

### Migration with `moved` blocks (recommended)

Add `moved` blocks in your root module for each Atlas-format region you used. Replace `module.atlas_aws` with your actual module name.

**PrivateLink endpoints** -- two `moved` blocks per region:

```terraform
moved {
  from = module.atlas_aws.mongodbatlas_privatelink_endpoint.this["US_EAST_1"]
  to   = module.atlas_aws.mongodbatlas_privatelink_endpoint.this["us-east-1"]
}

moved {
  from = module.atlas_aws.module.privatelink["US_EAST_1"]
  to   = module.atlas_aws.module.privatelink["us-east-1"]
}
```

**Encryption private endpoints** -- one `moved` block per region:

```terraform
moved {
  from = module.atlas_aws.module.encryption_private_endpoint["US_EAST_1"]
  to   = module.atlas_aws.module.encryption_private_endpoint["us-east-1"]
}
```

After adding the `moved` blocks:

1. Run `terraform plan` -- the plan should show no changes
2. Run `terraform apply` -- Terraform updates the state in-place
3. Remove the `moved` blocks from your configuration (Terraform ignores resolved `moved` blocks, but removing them keeps your code clean)

### Migration with `terraform state mv` (alternative)

If you prefer not to add `moved` blocks, use `terraform state mv` commands directly. Replace `module.atlas_aws` with your actual module name and `US_EAST_1` with each Atlas-format region you used:

```bash
terraform state mv \
  'module.atlas_aws.mongodbatlas_privatelink_endpoint.this["US_EAST_1"]' \
  'module.atlas_aws.mongodbatlas_privatelink_endpoint.this["us-east-1"]'

terraform state mv \
  'module.atlas_aws.module.privatelink["US_EAST_1"]' \
  'module.atlas_aws.module.privatelink["us-east-1"]'

terraform state mv \
  'module.atlas_aws.module.encryption_private_endpoint["US_EAST_1"]' \
  'module.atlas_aws.module.encryption_private_endpoint["us-east-1"]'
```

### Output key changes

Output map keys also change from Atlas format to AWS format:
- `module.atlas_aws.privatelink["US_EAST_1"]` becomes `module.atlas_aws.privatelink["us-east-1"]`
- `module.atlas_aws.privatelink_service_info["US_EAST_1"]` becomes `module.atlas_aws.privatelink_service_info["us-east-1"]`
- `module.atlas_aws.encryption.private_endpoints["US_EAST_1"]` becomes `module.atlas_aws.encryption.private_endpoints["us-east-1"]`

Update any references to these outputs in your configuration.

### Full example

Before (v0.2.x):

```terraform
module "atlas_aws" {
  source     = "terraform-mongodbatlas-modules/atlas-aws/mongodbatlas"
  version    = "~> 0.2"
  project_id = var.project_id

  privatelink_endpoints = [
    { region = "US_EAST_1", subnet_ids = [aws_subnet.east.id] },
    { region = "US_WEST_2", subnet_ids = [aws_subnet.west.id] },
  ]

  encryption = {
    enabled                  = true
    kms_key_arn              = aws_kms_key.atlas.arn
    private_endpoint_regions = ["US_EAST_1"]
  }
}

resource "aws_instance" "app" {
  # referencing output by Atlas-format key
  tags = {
    pl_status = module.atlas_aws.privatelink["US_EAST_1"].status
  }
}
```

After (v0.3.0):

```terraform
module "atlas_aws" {
  source     = "terraform-mongodbatlas-modules/atlas-aws/mongodbatlas"
  version    = "~> 0.3"
  project_id = var.project_id

  privatelink_endpoints = [
    { region = "US_EAST_1", subnet_ids = [aws_subnet.east.id] },
    { region = "US_WEST_2", subnet_ids = [aws_subnet.west.id] },
  ]

  encryption = {
    enabled                  = true
    kms_key_arn              = aws_kms_key.atlas.arn
    private_endpoint_regions = ["US_EAST_1"]
  }
}

resource "aws_instance" "app" {
  # output keys are now lowercase AWS format regardless of input format
  tags = {
    pl_status = module.atlas_aws.privatelink["us-east-1"].status
  }
}

# Temporary moved blocks -- remove after first successful apply
moved {
  from = module.atlas_aws.mongodbatlas_privatelink_endpoint.this["US_EAST_1"]
  to   = module.atlas_aws.mongodbatlas_privatelink_endpoint.this["us-east-1"]
}
moved {
  from = module.atlas_aws.module.privatelink["US_EAST_1"]
  to   = module.atlas_aws.module.privatelink["us-east-1"]
}
moved {
  from = module.atlas_aws.mongodbatlas_privatelink_endpoint.this["US_WEST_2"]
  to   = module.atlas_aws.mongodbatlas_privatelink_endpoint.this["us-west-2"]
}
moved {
  from = module.atlas_aws.module.privatelink["US_WEST_2"]
  to   = module.atlas_aws.module.privatelink["us-west-2"]
}
moved {
  from = module.atlas_aws.module.encryption_private_endpoint["US_EAST_1"]
  to   = module.atlas_aws.module.encryption_private_endpoint["us-east-1"]
}
```

The input `region` values can stay in either format. The module normalizes internally. Only the `for_each` keys (state addresses) and output map keys change.

## Private Endpoint Regional Mode (Breaking Change)

v0.3.0 adds a `privatelink_regional_mode` variable (`"auto"` or `"disabled"`, default `"disabled"`). The module creates `mongodbatlas_private_endpoint_regional_mode` only when the value is `"auto"` and the configuration has multiple distinct Atlas service regions (see [regionalized private endpoints for multi-region sharded clusters](https://www.mongodb.com/docs/atlas/security-private-endpoint/?cloud-provider=aws#-optional--regionalized-private-endpoints-for-multi-region-sharded-clusters)).

In v0.2.x, the module turned on private endpoint regional mode automatically whenever it detected more than one Atlas region.

### Why the change?

Some deployments need multiple regions or cross-region PrivateLink without per-region SRV records or connection strings. Because project-wide regional mode also has operational impact, we now use an opt-in default.

### Who is affected?

- Users with **more than one distinct Atlas service region** in `privatelink_endpoints` or `privatelink_byo_endpoint` who **relied on automatic regional mode** in v0.2.x.
- Anyone who reads `regional_mode_enabled` and assumed it was `true` whenever multiple service regions were present (it is now `true` only when `privatelink_regional_mode = "auto"` and multiple regions exist).

**No action** if you use a single Atlas region for PrivateLink, or you use only cross-region VPC endpoints that point at one primary service region.

### Required action

To keep the same behavior as v0.2.x (create `mongodbatlas_private_endpoint_regional_mode` when the module would have done so before), set:

```terraform
privatelink_regional_mode = "auto"
```

Add this to your `module` block next to your other `privatelink_*` settings.

### Plan after upgrade

If you  **do not** set `privatelink_regional_mode = "auto"` and you previously had multiple Atlas service regions, `terraform plan` may show that `mongodbatlas_private_endpoint_regional_mode` will be **destroyed**. That is expected: the resource is no longer created unless you opt in. Confirm that your applications and connection strings do not require per-region SRV before you apply. If they do, set `privatelink_regional_mode = "auto"` before applying.

## Configurable Timeouts

v0.3.0 introduces a `timeouts` variable with `create`, `update`, `delete` fields defaulting to `"30m"`. The module applies these timeouts to all wrapped Atlas and AWS resources.

### Who is affected?

**Existing deployments upgrading from v0.2.x** will see a plan diff: the new `"30m"` defaults emit timeout blocks that did not previously exist. The diff is an in-place update, not destructive -- no resources are recreated.

### Zero-diff upgrade

Set `timeouts = null` to skip all module-managed timeout blocks and keep provider defaults:

```terraform
module "atlas_aws" {
  source     = "terraform-mongodbatlas-modules/atlas-aws/mongodbatlas"
  version    = "~> 0.3"
  project_id = var.project_id

  timeouts = null

  # ... rest of configuration
}
```

After the upgrade, you can remove `timeouts = null` to opt into the `"30m"` defaults, or replace it with custom values (e.g., `timeouts = { create = "45m", update = "45m", delete = "15m" }`).

After upgrading:
1. Run `terraform plan` with `timeouts = null`. The plan should show no timeout-related changes.
2. Run `terraform apply`.

### Post-import usage

`timeouts = null` is also useful after `terraform import`. Imported resources have no timeout blocks in state, so the `"30m"` defaults would create a plan diff. Setting `timeouts = null` avoids this.

## S3 Versioning Default Changed

The default value of `versioning_enabled` changed from `true` to `false` on both `backup_export.create_s3_bucket` and `log_integration.create_s3_bucket`.

### Why the change?

Atlas writes timestamp-based object keys with no overwrite risk, so versioning adds storage cost without benefit. S3 lifecycle expiration rules also do not delete previous object versions on versioned buckets; they only add delete markers. Disabling versioning ensures lifecycle rules work as expected and avoids accumulating non-current versions that never expire.

### Who is affected?

Users with module-managed S3 buckets who relied on the previous `versioning_enabled = true` default.

### Required action

Set `versioning_enabled = true` explicitly to preserve the previous behavior:

```terraform
backup_export = {
  enabled = true
  create_s3_bucket = {
    enabled            = true
    versioning_enabled = true
  }
}
```

### Zero-diff option

If you previously accepted the default (`versioning_enabled = true`) and want to keep versioning enabled, add `versioning_enabled = true` explicitly before upgrading. The plan shows no changes.

## S3 Lifecycle Rules Added

`expiration_days` is a new field on `create_s3_bucket` (default: 365 days for backup export, 90 days for log integration). A new `aws_s3_bucket_lifecycle_configuration` resource appears in the plan.

### Who is affected?

Users with module-managed S3 buckets. The plan shows a new `aws_s3_bucket_lifecycle_configuration` resource.

### Required action

No action required if the lifecycle rules are acceptable. To opt out, set `expiration_days = 0`:

```terraform
backup_export = {
  enabled = true
  create_s3_bucket = {
    enabled         = true
    expiration_days = 0
  }
}
```

### Zero-diff option

Set `expiration_days = 0` before upgrading. No lifecycle configuration resource appears in the plan.

## Bucket Name Validation (Dots Disallowed)

`create_s3_bucket.name` and `name_prefix` now reject dot (`.`) characters. Dots in S3 bucket names are incompatible with the virtual-hosted-style addressing that [Data Exfil Prevention](https://www.mongodb.com/docs/atlas/security/restrict-data-access-by-network/) requires.

### Who is affected?

Users with module-managed S3 buckets whose names contain dots.

### Required action

To replace bucket names with dots, complete the following steps:

1. Create a new bucket without dots.
2. Copy objects from the old bucket to the new bucket.
3. Update the module configuration to use the new bucket name.
4. Remove the old bucket.

Alternatively, switch to a BYO bucket (`bucket_name`) to manage naming independently.

## BYOE Variables Changed

The name of BYOE PrivateLink variables changed. The type of `privatelink_byo_endpoint` changed from `map(string)` to `map(object({ region, supported_remote_regions }))` to support cross-region PrivateLink.

The following table compares the previous names of the BYOE PrivateLink variables (v0.2.x) against the new names (v0.3.0):

| v0.2.x variable | v0.3.0 variable |
|---|---|
| `privatelink_byoe_regions` | `privatelink_byo_endpoint` |
| `privatelink_byoe` | `privatelink_byo_service` |

### Who is affected?

Users of the BYOE PrivateLink pattern upgrading from v0.2.x.

### Required action

Rename the variables and wrap each region string in an object with a `region` field:

Before (v0.2.x):

```terraform
module "atlas_aws" {
  source     = "terraform-mongodbatlas-modules/atlas-aws/mongodbatlas"
  version    = "~> 0.2"
  project_id = var.project_id

  privatelink_byoe_regions = {
    east = "us-east-1"
    west = "us-west-2"
  }

  privatelink_byoe = {
    east = {
      vpc_endpoint_id = aws_vpc_endpoint.atlas_east.id
    }
    west = {
      vpc_endpoint_id = aws_vpc_endpoint.atlas_west.id
    }
  }
}
```

After (v0.3.0):

```terraform
module "atlas_aws" {
  source     = "terraform-mongodbatlas-modules/atlas-aws/mongodbatlas"
  version    = "~> 0.3"
  project_id = var.project_id

  privatelink_byo_endpoint = {
    east = {
      region = "us-east-1"
    }
    west = {
      region                   = "us-west-2"
      supported_remote_regions = ["us-east-1"]
    }
  }

  privatelink_byo_service = {
    east = {
      vpc_endpoint_id = aws_vpc_endpoint.atlas_east.id
    }
    west = {
      vpc_endpoint_id = aws_vpc_endpoint.atlas_west.id
    }
  }
}
```

Set `supported_remote_regions` only for regions that accept cross-region VPC endpoint connections. Omit it (or pass `[]`) for same-region-only endpoints.
